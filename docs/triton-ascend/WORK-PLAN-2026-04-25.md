**Compiler-level work? YES** — non-trivial:
1. **LLVM hash bump**: `7d5de303` → `f6ded0be`. AscendNPU-IR and triton-ascend's
   C++ compiler code targets a specific LLVM/MLIR API version; LLVM bump
   means API drift risk in `lib/` headers (`mlir/IR/*`, `mlir/Dialect/*`
   signatures may have shifted).
2. **`lib/Dialect/` churn** in community triton: 85 files / ~9k inserts
   including `TritonNvidiaGPU` transforms (PlanCTA, TMALowering). These
   touch shared MLIR dialect definitions that triton-ascend's NPU
   transforms may use.
3. **Backend abstract API changed**:
   `BaseBackend.get_arg_specialization(arg, ty, **kwargs)` →
   `get_int_specialization(arg, **kwargs)`. But this is Python; the
   vendored `python/triton` rebase takes this automatically, no NPU
   backend action (NPU backend doesn't override it per grep).

**Non-compiler work** (python/triton rebase layer):
- 144 `python/*.py` files changed. Vendored community code — rebase
  take-upstream by default; conflicts only where NPU hooked in
  (runtime/jit.py maybe; need to check actual diff).

## Prerequisites for doing the port

1. **LLVM prebuild** matching `f6ded0be`. triton-ascend's build system
   requires this; without it C++ compile fails. Check
   `bin/build-llvm.sh` and `third_party/ascend/llvm_patch/`.
2. **AscendNPU-IR compatibility** with v3.6.0's MLIR. If AscendNPU-IR
   targets the old LLVM hash, it may need its own bump first (or the
   existing tip `de20e14` may already target new LLVM — need to check
   AscendNPU-IR's own LLVM-version note).
3. **Bisheng / CANN toolchain** on A3 for building + running smoke.

## Honest risks

1. **Compiler C++ rebase is NOT a task for a fresh LLM cold-drive**.
   It needs LLVM/MLIR domain expertise + build iteration on A3. The
   skill's KB documents the conflict surfaces but can't shortcut the
   actual C++ API fixes.
2. **AscendNPU-IR may need its own bump first** (we pinned at
   `1b33649`; master is at `de20e14`). If AscendNPU-IR hasn't yet
   tracked v3.6.0 LLVM, triton-ascend's compile fails against that.
   Chicken-and-egg: check AscendNPU-IR's own status before rebase.
3. **CANN bundle compatibility**: triton-ascend's NPU kernels might
   expect specific CANN runtime symbols. A CANN bump may be needed
   alongside.
4. **Not end-to-end 0-interaction testable today**: unlike vllm-ascend
   / torch_npu / transformers port-experts, this one crosses into C++
   compiler work. Cold-drive can get to "rebase attempted, conflicts
   classified" but can't auto-fix MLIR API drift.
