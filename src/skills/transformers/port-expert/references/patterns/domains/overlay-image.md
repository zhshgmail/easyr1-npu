# Domain — Dockerfile.overlay (pip-install target transformers onto existing image)

**Load when**: Phase C building the Day-0 overlay image (outcome A or B).

## Why "overlay" instead of rebuild

Rebuilding from `FROM quay.io/ascend/verl:*` + all deps adds ~10-15min
and re-triggers NPU-BUG-001 / NPU-BUG-004 workarounds. Since we're only
changing one dep (transformers), overlay is dramatically faster:

- Outcome A (works-as-is): overlay = `FROM base-image + pip install transformers==X`.
  Build time ~30-60s.
- Outcome B (forward-port): overlay = above + `COPY patches/npu_flash_attention.py /path/in/site-packages/`.
  Still ~60s.
- Outcome C: no build; this file doesn't apply.

## Dockerfile.overlay template (outcome A)

```dockerfile
# Overlay image: BASE_IMAGE + target transformers release.
# Generated by transformers-day0-expert for TARGET_TRANSFORMERS_VERSION.

ARG BASE_IMAGE=<BASE_IMAGE tag>
FROM ${BASE_IMAGE}

ENV DEBIAN_FRONTEND=noninteractive
ENV PIP_ROOT_USER_ACTION=ignore
ENV PIP_INDEX_URL=https://mirrors.aliyun.com/pypi/simple/
ENV PIP_DEFAULT_TIMEOUT=60

# Install the target community transformers release.
# --no-deps: don't let pip re-pull half the torch stack; we trust the
#   base image's stack. Then manually pin any direct deps target needs
#   that aren't already present (rare; verify via Phase A scan).
RUN pip install --no-cache-dir --no-deps transformers==<TARGET_VERSION>

# Optional: if pip warned about a direct dep that's genuinely needed and
# missing (e.g. 'regex' bump, 'tokenizers' bump), add it here.
# RUN pip install --no-cache-dir tokenizers==<newer>

# Sanity (build-time — fast-fail). VERSION-ONLY, do NOT import
# npu_flash_attention here. See §"Build-time import trap" below.
RUN TORCH_DEVICE_BACKEND_AUTOLOAD=0 python3 -c '
import transformers
print("overlaid transformers:", transformers.__version__)
'
```

**Build-time import trap (2026-04-23 day0 finding)**: it is tempting to
add `from transformers.integrations import npu_flash_attention` inside
the Dockerfile sanity block. **Don't.** That import chain triggers
`torch_npu → libascend_hal.so` dlopen, which fails in the docker build
sandbox (no NPU device mount). The build aborts with a misleading
"torch_npu backend failed to load" error.

Only verify the `transformers` version at build time. Test
`from transformers.integrations import npu_flash_attention` at **runtime**,
inside a smoke container that actually has NPU devices mounted. If it
fails there, it's a real adapter problem; if you catch it at build time,
it's a false alarm from the missing device mount.

First observed 2026-04-23 during transformers-day0 wet-run on 5.6.0
(session `trans-day0-wetrun-20260423-0109` iter1 build failure). Fix
landed as iter2 of that session's Dockerfile.

Image tag convention: `easyr1-npu-trans<major><minor>:<SESSION_TAG>`
(e.g. `easyr1-npu-trans56:trans-day0-20260423-0103`).

## Dockerfile.overlay template (outcome B — forward-port)

```dockerfile
ARG BASE_IMAGE=<BASE_IMAGE tag>
FROM ${BASE_IMAGE}

ENV DEBIAN_FRONTEND=noninteractive
ENV PIP_ROOT_USER_ACTION=ignore
ENV PIP_INDEX_URL=https://mirrors.aliyun.com/pypi/simple/
ENV PIP_DEFAULT_TIMEOUT=60

RUN pip install --no-cache-dir --no-deps transformers==<TARGET_VERSION>

# Forward-port patch: override site-packages npu_flash_attention.py with
# our adapter that handles the new ALL_ATTENTION_FUNCTIONS keys / new
# kwargs.
COPY patches/npu_flash_attention.py \
     /usr/local/python3.11.14/lib/python3.11/site-packages/transformers/integrations/npu_flash_attention.py

# Sanity check at BUILD time: ONLY verify the patch file was written and
# transformers itself still imports. Do NOT import npu_flash_attention
# here — that triggers torch_npu → libascend_hal.so dlopen, which fails
# in the docker-build sandbox (no NPU devices). See §"Build-time import
# trap" above.
RUN TORCH_DEVICE_BACKEND_AUTOLOAD=0 python3 -c '
import transformers, os
print("overlaid transformers:", transformers.__version__)
patch_path = "/usr/local/python3.11.14/lib/python3.11/site-packages/transformers/integrations/npu_flash_attention.py"
assert os.path.isfile(patch_path), f"patch not at {patch_path}"
print("patch file in place")
'

# Runtime patch verification happens in the FIRST smoke container (V1.1)
# where NPU devices are mounted. At that point, `import npu_flash_attention`
# + a registry-key check is meaningful. Don't try to shift it into the
# Dockerfile.
```

The `patches/npu_flash_attention.py` source lives in `$WORKSPACE/patches/`
during the session. Origin: cherry-picked from base image's copy + new
handler logic added. Record the diff in `$WORKSPACE/forward-port-diff.patch`.

## Build command (runs on A3 via deploy_to_a3.sh)

```
bash $TRANSFORMERS_DAY0_EXPERT_ROOT/scripts/deploy_to_a3.sh \
    --branch ascend-day0-trans-$SESSION_TAG \
    --image-tag easyr1-npu-trans<MM>:<SESSION_TAG> \
    --dockerfile Dockerfile.overlay-trans<MM> \
    --upstream-consumer $UPSTREAM_CONSUMER \
    --base-image <BASE_IMAGE>
```

(Adjust deploy_to_a3.sh's `--base-image` support if it's not already
there; the Stage-2 sibling scripts accept it.)

## Post-build verification

Before Phase D smoke, verify overlay image:

```
docker run --rm <overlay-image> python3 -c '
import torch, torch_npu, transformers, vllm
print("all imports OK")
print("transformers:", transformers.__version__)
print("expect:", "<TARGET_VERSION>")
'
```

If import fails: the overlay broke a dep chain the base image relied on.
Likely a forgotten `--no-deps` or a transitive dep version collision.
Fix Dockerfile.overlay, rebuild.

## Rollback / cleanup

On session exit, `cleanup_session.sh --session-tag <TAG>
--keep-user-provided <base-image-tag>` removes the overlay image but
keeps the base. Overlay images are cheap (~1-2GB delta per transformers
install), but do accumulate — keep cleanup tight.
