# triton-ascend port-expert KB — Search Index

## Drift model

triton-ascend is a **fork** of community triton, not a plugin that
imports from it. Upgrade = rebase-onto-new-tag + resolve conflicts.
There is no F1–F8 symbol drift taxonomy here; the drift shape is "git
merge conflicts". KB records the **conflict surfaces** and
**resolution recipes** seen in real rebases.

## Repo layout (verified on triton-ascend 3.5.0, 2026-04-25)

- `python/triton/**` — **vendored community triton Python source**.
  Primary rebase-conflict surface. Changes here are almost always
  take-upstream, except where NPU hooks in (see below).
- `python/src/` — pybind glue between Python and MLIR/triton C++.
  `ir.cc` / `ir.h` are the hotspots.
- `include/triton/Dialect/Triton/IR/**` — shared MLIR dialect TD
  files. Both community and NPU add enum cases here; watch for
  **enum index collisions**.
- `third_party/ascend/backend/` — **NPU backend Python source**
  (the real one; `python/triton/backends/ascend/` is generated by
  `setup.py` at install time).
- `third_party/ascend/lib/` + `third_party/ascend/include/` — 149 files
  of NPU-specific C++/MLIR compiler code. Expose LLVM/MLIR API drift
  when community bumps LLVM hash.
- `third_party/ascend/llvm_patch/<hash>.patch` — NPU's custom LLVM
  patch. triton-ascend pins its own LLVM commit (e.g. `fad32722`)
  plus a patch file applied on top. Bumping LLVM base is a **separate
  multi-week task** independent of the triton rebase.
- `third_party/ascend/AscendNPU-IR/` — git submodule with 60+ Huawei
  MLIR patches. Own release cadence; may need bump in parallel.
- `version.txt` + `python/triton/__init__.py:__version__` + `setup.py`'s
  `TRITON_VERSION` constant — current community-triton baseline.

## Community triton upstream URL

Not in `origin` remote; `origin` goes to `gitcode.com/Ascend/triton-ascend`.
Add community as a second remote:

```bash
git remote add community https://github.com/triton-lang/triton.git
git fetch community --tags
```

## The FIRST rebase decision: merge, not rebase-onto

**DO NOT** attempt `git rebase --onto <community-tag> v<old-tag> origin/main`.
triton-ascend's `origin/main` is **not** a descendant of any community
tag. Check:

```bash
git merge-base --is-ancestor v3.5.0 origin/main
echo $?  # 1 = NOT ancestor, confirms vendored-fork pattern
```

Instead, use **`git merge <target-tag> --no-commit --no-ff`**. Git's
3-way merge uses the nearest common ancestor and correctly identifies
community-already-applied commits vs NPU-specific commits.

## Conflict surfaces (verified 2026-04-25 on v3.5.0 → v3.6.0, 12 conflict files)

### Trivial (resolve by selection, seconds each)

| File | Conflict | Resolution |
|---|---|---|
| `.gitignore` | `.idea/` (NPU) vs `.cursor` (community) | Keep both |
| `python/triton/__init__.py` | `__version__` string | Take target-tag string |
| `setup.py` | `TRITON_VERSION` + `TRITON_WHEEL_VERSION_SUFFIX` env var plumbing | Take community — its `get_triton_version_suffix()` already handles the env var in v3.6.0 |
| `README.md` | Entire file: NPU markdown vs community markdown | Keep NPU (the project IS NPU-focused) |
| `Makefile` | NPU test-unit/test-mlir/test-inductor targets vs community's CUDA/gluon/proton targets | Keep NPU (community targets assume CUDA) |
| `docs/conf.py`, `docs/getting-started/installation.rst` | NPU DELETED these; community modified | `git rm` (honor NPU's deletion intent) |

### Compiler-level (require judgment)

| File | Conflict | Resolution recipe |
|---|---|---|
| `cmake/llvm-hash.txt` | NPU pins patched LLVM (e.g. `fad32722`) vs community pins unpatched LLVM | **Keep NPU** (community LLVM won't apply `third_party/ascend/llvm_patch/*.patch`). Risk: community's C++ may call new LLVM APIs that old LLVM base doesn't have. Only discoverable at compile time; may force an LLVM-bump follow-up task. |
| `include/triton/Dialect/Triton/IR/TritonAttrDefs.td` | MLIR enum index collision: both sides assigned the same index to different values | Merge all enum cases; reassign NPU's case to a non-conflicting index. **Backward-compat risk**: pre-existing serialized MLIR IR with the old index meaning now means a different enum case. Document for downstream. |
| `python/src/ir.cc` | pybind exports + `MLIRContext` constructor | InputPrecision: merge all enum values in pybind. MLIRContext: keep community's `Threading::DISABLED` init, ADD NPU's `__enter__`/`__exit__` context-manager (non-conflicting). |
| `python/src/ir.h` | `TritonOpBuilder::restoreInsertionPoint` + private members | Take community's `restoreInsertionPoint` (more robust edge-case handling via `getLocForBlock`); in private section, take community's `mlir::`-qualified members + `getLocForBlock` helper, KEEP NPU's `compile_mode` field. |

### Interpreter (the biggest python-layer conflict)

| File | Conflict | Resolution recipe |
|---|---|---|
| `python/triton/runtime/interpreter.py` | NPU adds Ascend-specific interpreter builder + sub-vec execution branch; community refactors arg-binding + adds `try/finally` around grid loop | Take community's outer structure (arg-binding, grid setup, `try/finally` with `patch_scope.restore()`). Inside the inner `try`, branch on `hasattr(interpreter_builder, 'execute_with_sub_vec_simulation')`: NPU path takes that call, else-branch keeps the per-(x,y,z) grid loop. |

## Verification status after merge

- **Text-level merge**: all 12 conflicts resolved, `git commit` clean.
- **`py_compile`** of all Python files touched: PASS.
- **`py_compile`** of 12 NPU backend Python files (`third_party/ascend/backend/*.py`): PASS.
- **`import triton`** at Python level: **FAILS** at `from triton._C.libtriton import getenv` — needs compiled C extension. **This is the signal that text-level merge is only half the work.**
- **C++ build**: NOT ATTEMPTED this session (needs LLVM prebuild + A3 hardware).

## Remaining work after text-merge

1. **LLVM API drift check**: does `fad32722` LLVM base provide all MLIR
   APIs that community v3.6.0 `lib/` uses? If not, either (a) port the
   community change to old LLVM APIs, or (b) bump LLVM base +
   re-patch. Only discoverable via build.
2. **AscendNPU-IR version sync**: triton-ascend v3.6.0 may require a
   specific AscendNPU-IR commit; submodule pin may need update.
3. **CANN runtime compat**: NPU kernels in `third_party/ascend/lib/`
   link against CANN. Check if CANN version on A3 matches.
4. **Build + test cycle** on A3. Inside a NPU container with
   `TRITON_CODEGEN_BACKENDS=ascend` and CANN installed. Run
   `python/test/unit/ascend/` if present.
5. **Smoke NPU JIT**: tiny `@triton.jit` kernel on NPU device, e.g.
   `vector_add(x, y, out)`.

## What is NOT this skill's job

- Fixing C++/MLIR compile errors after LLVM API drift — AIL team territory.
- Bumping LLVM base + regenerating patch — separate task with its own review process.
- CANN driver / kernel library compatibility — torch_npu territory overlaps.
- Performance regressions — separate benchmark/optimization task.

## Relationship to other port-experts

- **torch_npu upgrade may require newer triton-ascend** — if torch bumps
  a `triton>=3.M` constraint, check `version.txt` and trigger this
  skill if needed. Run torch-npu and triton-ascend rebases in parallel
  when both bumps land in the same CANN cycle.
- **vllm-ascend → triton-ascend**: no direct rebase coupling. vllm-ascend's
  triton usage is through torch_npu's inductor path; API changes
  surface as F3 in vllm-ascend's scanner if at all.
- **transformers → triton-ascend**: no path.
